# 📘 Master Architecture Document: AI Matching & Recommendation Engine (V1)

## 🏢 1. Architecture Blueprint

### 1.1 Core Architectural Principles
*   **Semantic AI (Vector Search):** We abandon rigid keyword matching (e.g., "Python" == "Python") in favor of Large Language Model (LLM) embeddings. The AI understands intent (e.g., matching "data visualization" with "Tableau expert").
*   **Event-Driven Read Model (CQRS):** This microservice does not own original user data. It listens to ecosystem events and builds a highly optimized, flattened "Read Replica" containing only the data necessary for matchmaking.
*   **V1 Pragmatism (`pgvector`):** Instead of paying for a dedicated Graph Database on Day 1, we utilize the `pgvector` extension inside our existing Cloud SQL instance. This handles 90% of the matchmaking value (Semantic Bio matching) for $0 extra infrastructure cost.
*   **Continuous Learning:** The engine actively logs positive interactions (successful sessions) and negative interactions (user dismisses a recommendation) to dynamically adjust future SQL matching scores.

### 1.2 The Technology Stack (Google Cloud Platform)
*   **API Gateway:** **Google Cloud API Gateway** + **Cloud Armor** (WAF).
*   **Application Server:** **Google Cloud Run** (Serverless containers).
*   **Framework:** **FastAPI (Python)** using SQLAlchemy 2.0.
*   **Database:** **Cloud SQL for PostgreSQL** *(Logically isolated as `db_matching`)*. Requires `CREATE EXTENSION vector;`.
*   **AI Engine:** **Google Vertex AI (Text Embeddings API)**.
*   **Message Broker:** **Google Cloud Pub/Sub**.

## 🗄️ 2. Database Schema Design (Cloud SQL + pgvector)

Since this is an isolated Read-Model, the schema is incredibly flat and optimized purely for mathematical similarity searches.

### 2.1 The Vector Store
*   **`match_profiles`** *(The Semantic Brain)*
    *   `user_id` (UUID, Primary Key) - *Foreign reference to Global Identity.*
    *   `role` (ENUM: `MENTEE`, `MENTOR`, `BOTH`)
    *   `is_active` (BOOLEAN, Default: TRUE) - *Filters out paused mentors.*
    *   `combined_text_payload` (TEXT) - *A concatenated string of the user's bio, learning goals, and expertise (e.g., "High school junior wanting to learn Python and robotics...").*
    *   `embedding` (**VECTOR(768)**) - *The 768-dimensional mathematical array generated by Google Vertex AI.*
    *   `updated_at` (TIMESTAMPTZ)
    *   🚀 **Index:** `CREATE INDEX ON match_profiles USING ivfflat (embedding vector_cosine_ops);` *(Optimizes vector search to sub-millisecond speeds).*

### 2.2 The Feedback Loop (Simulating Graph Edges)
Because we aren't using Neo4j yet, we use this table to mimic basic graph relationships (Affinity & Rejection) directly in Postgres.
*   **`match_interactions`**
    *   `interaction_id` (UUID, Primary Key)
    *   `source_user_id` (UUID, Indexed) - *The Mentee looking at recommendations.*
    *   `target_user_id` (UUID, Indexed) - *The Mentor being recommended.*
    *   `interaction_type` (ENUM: `REJECTED_SUGGESTION`, `SUCCESSFUL_MENTORSHIP`)
    *   `weight` (INTEGER) - *e.g., `-100` for rejected, `+50` for successful mentorship.*
    *   `created_at` (TIMESTAMPTZ)
    *   *Constraint:* `UNIQUE(source_user_id, target_user_id, interaction_type)`

## 🌐 3. REST API Design (FastAPI)

### 3.1 Public Recommendation APIs (Called by Client SPA)
*   **`GET /api/v1/recommendations/mentors`** 🏆 *(The Core AI Endpoint)*
    *   *Headers:* `X-User-Id: usr_123` (The Mentee).
    *   *Logic:* 
        1. Fetch Mentee's embedding vector from `match_profiles`.
        2. Execute an optimized `pgvector` Cosine Distance (`<=>`) SQL query against all Mentor embeddings.
        3. `LEFT JOIN` the `match_interactions` table to completely exclude any `target_user_id` the mentee previously `REJECTED_SUGGESTION`.
    *   *Returns (200 OK):* 
        ```json
        {
          "recommendations":[
            {"mentor_id": "mtr_99", "match_score_percentage": 94},
            {"mentor_id": "mtr_42", "match_score_percentage": 89}
          ]
        }
        ```

*   **`POST /api/v1/recommendations/feedback`**
    *   *Purpose:* Allows the youngster to "Swipe Left" (Hide) a recommendation.
    *   *Body:* `{"target_user_id": "mtr_99", "interaction_type": "REJECTED_SUGGESTION"}`
    *   *Returns:* `201 Created`. (Engine guarantees it won't suggest this mentor to this mentee again).

## 📡 4. Ecosystem Choreography (Pub/Sub Events)

The AI Engine relies entirely on Pub/Sub to build its knowledge base.

### 4.1 Consumed Events (Building the Vector Brain)
**Topic: `profiles.events`**
*   **Event:** `MENTEE_PROFILE_UPDATED` or `MENTOR_PROFILE_UPDATED`
*   **Payload:** `{"user_id": "usr_123", "role": "MENTOR", "bio": "I teach AI...", "skills": ["Python"]}`
*   **Action:** 
    1. FastAPI concatenates the text: `"Role: Mentor. Bio: I teach AI... Skills: Python"`.
    2. FastAPI sends the string to **Google Vertex AI Text Embeddings API**.
    3. Vertex AI returns `[0.14, -0.05, 0.99...]`.
    4. FastAPI performs an `UPSERT` on the `match_profiles` PostgreSQL table with the new vector.

**Topic: `mentoring.sessions`**
*   **Event:** `SESSION_COMPLETED`
*   **Payload:** `{"mentor_id": "mtr_1", "mentee_id": "mte_1", "rating": 5}`
*   **Action:** If rating >= 4, inserts a `SUCCESSFUL_MENTORSHIP` row into `match_interactions`. (This creates a positive bias for future algorithmic tuning).

## 🔄 5. Master Workflows

### Workflow A: Asynchronous Knowledge Ingestion (The AI Learns)
1.  **Action:** A mentor updates their expertise area in the Mentor-Mentee App.
2.  **Broadcast:** The Mentor Service publishes `MENTOR_PROFILE_UPDATED`.
3.  **LLM Processing:** The AI Matching FastAPI service consumes the event. It makes an internal gRPC/REST call to Google Vertex AI.
4.  **Vector Storage:** Vertex AI translates the English text into a mathematical vector. FastAPI updates the `db_matching` PostgreSQL database. 
5.  *Result:* The mentor is instantly indexed and ready to be discovered by mentees seeking those specific skills.

### Workflow B: The Semantic Recommendation Request (The Matchmaker)
1.  **Intent:** A youngster opens the Dashboard SPA. Widget B ("AI Matchmaker") is loading.
2.  **The API Call:** The Dashboard calls `GET /api/v1/recommendations/mentors` via the Gateway.
3.  **The SQL Vector Math:** FastAPI executes this exact SQL logic in Cloud SQL:
    ```sql
    SELECT target.user_id, 
           1 - (target.embedding <=> mentee.embedding) AS match_score
    FROM match_profiles target, match_profiles mentee
    WHERE mentee.user_id = 'usr_123'
      AND target.role = 'MENTOR'
      AND target.is_active = TRUE
      AND target.user_id NOT IN (
          SELECT target_user_id FROM match_interactions 
          WHERE source_user_id = 'usr_123' AND interaction_type = 'REJECTED_SUGGESTION'
      )
    ORDER BY target.embedding <=> mentee.embedding
    LIMIT 5;
    ```
4.  **The Resolution:** FastAPI returns the top 5 Mentor IDs with an exact semantic match percentage. The UI seamlessly renders the recommendation carousel.

### Workflow C: The Feedback Loop (The AI Self-Corrects)
1.  **Action:** The AI suggests an "Accounting Mentor" to a youngster who wants to learn "FinTech" (Vector similarity was close, but the youngster finds it boring).
2.  **Rejection:** The youngster clicks the "X" (Not Interested) on the UI widget.
3.  **Database Update:** FastAPI logs a `REJECTED_SUGGESTION` row in `match_interactions`.
4.  **Result:** The SQL query in Workflow B mathematically guarantees that this specific mentor will never appear on this mentee's dashboard again.
